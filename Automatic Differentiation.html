In today's post, I want to talk about what automatic differentiation is and give you a simple method for implementing automatic differentiation in your code. Today's post will use Python as the language of choice, but everything said here generalizes to other programming languages. Automatic differentiation is a way to efficiently and accurately calculate derivatives of functions from their computer code, but before we get to automatic differentiation, let's talk about why you should want to compute derivatives in the first place!

The Utility of Derivatives¶

The Newton Method¶

The first example we will use to illlustrate why derivatives are useful for many numerical algorithms is the Newton method. The Newton method is a simple iterative technique for estimating the root of a function. Given a function $f(x)$ (which may be nonlinear or quite complciated), we want to find a value for $x^\ast$ that is a root, meaning that $f(x^\ast) = 0$.

The Newton method is simple to derive. It's based on a Taylor series expansion. Near a point $a$, we can approximate $f(x)$ as a linear polynomial given the value of $f(a)$ and the derivative at that point $f'(a)$: $$ f(x) \approx f(a) + f'(a)(x-a) $$ Suppose we can at least make a guess $\hat{x}$ for the root $x^\ast$. If our guess isn't too bad, it'll be close to $x^\ast$. We can use the linear polynomial approximation from the Taylor series to "measure" how bad our guess is. If $\hat{x}$ was correct, then $f(\hat{x}) = 0$ and we'd have found our root. Based on the sign of $f(\hat{x})$ and the sign of $f'(\hat{x})$, we can figure out which direction the root is. And we can compute a correction to our guess. Basically you plug $\hat{x}$ into the Taylor polynomial and solve for $x^\ast$. $$ f(x^\ast) = 0 \approx f(\hat{x}) + f'(\hat{x})(x^\ast - \hat{x}) $$ Which means that $$ x^\ast \approx \hat{x} -\frac{f(\hat{x})}{f'(\hat{x})} $$ Note that since the Taylor polynomial is not an exact representation of $f$, we don't get an exact solution by solving it. But we can prove that it's a better guess if we consider the higher order terms. First, let's compute how about our initial guess, then how bad the new guess is. $$ \begin{align} 0 &= f(\hat{x}) + f'(\hat{x})(x^\ast - \hat{x}) + \frac{1}{2}f''(\hat{x})(x^\ast - \hat{x})^2 + \cdots\\ 0 &= f(\hat{x}) + f'(\xi)(x^\ast - \hat{x})\\ \frac{-f(\hat{x})}{f'(\xi)} &= x^\ast - \hat{x} = \varepsilon \end{align} $$ The second step follows from the Mean Value Theorem, where the remainder depends on the error $\varepsilon = x^\ast - \hat{x}$ and is equal to $f'(\xi)\varepsilon $ for some value $\xi \in [x^\ast,\hat{x}]$. This means that the error in our guess is proprortional to how much $f(\hat{x}$ differs from zero. If the function is changing a lot around it's root, then this error won't be very large because $f'(x)$ is large and therefore it's reciprocal is small. Now if we update our guess using Newton's method, what's the error? $$ \begin{align} 0 &= f(\hat{x}) + f'(\hat{x})(x^\ast - \hat{x}) + \frac{1}{2}f''(\hat{x})(x^\ast - \hat{x})^2 + \cdots\\ 0 &= f(\hat{x}) + f'(\hat{x})(x^\ast - \hat{x}) + \frac{1}{2}f''(\xi)(x^\ast - \hat{x})^2 \\ \hat{x} -\frac{f(\hat{x})}{f'(\hat{x})} &= x^\ast + \frac{1}{2}\frac{f''(\hat{\xi})}{f'(\hat{x})}(x^\ast - \hat{x})^2 \\ \frac{1}{2}\frac{f''(\xi)}{f'(\hat{x})}(x^\ast - \hat{x})^2 &= \varepsilon' \end{align} $$ The remainder shows that the error of the new guess is proportional to the square of the error in the old guess. For guesses that start close enough to the correct value, the square of the error is smaller. Also if the curvature of the function (as determined by the second derivative $f''$) is small near the root, then the error after an update will be very small.

Under certain conditions (which we won't derive here), repeating this process over and over will cause our guesses to get better and better. We generate a sequence of guesses $\hat{x}_0, \hat{x}_1, \hat{x}_2, \hat{x}_3, \ldots$ where each one is calculated from the last using the update step:

$$ \hat{x}_{n+1} = \hat{x}_{n} - \frac{f(\hat{x}_n)}{f'(\hat{x}_n)} $$

In fact, we can make any guess as good as we want. And that's exactly what a limit) is! So in the limit of infinitely many of iterations, this estimate converges.

$$ \lim_{n\to\infty} \hat{x}_n = x^\ast $$

Let's look at a numerical example, where we calculate $\sqrt{2}$. Our function is the polynomial $f(x) = x^2 - 2$ and it's derivative is $f'(x) = 2x$.

You can see that the Newton method converges very quickly! With a 3-4 iterations, we get an estimate that's accurate to 5-10 decimal places. Under the right conditions, the rate of convergence is quadratic. Intuitively that means every iteration doubles the correct number of decimal places. In fact, this is exactly the method that many calculators or computers use when you ask them to calculate $\sqrt{x}$. They apply Newton's method to $f(y) = y^2 - x$ and return the $y$ after the error goes down below the calculators' accuracy.

It's hard to beat Newton's method for speed and it works for higher dimensional functions too where the linearized version is $f(x) \approx f(a) + Df(a)(x-a)$ and the derivative is now a square matrix $Df(a)$ (called the Jacobian). So the update for the multidimensional version involves solving a linear system of equations (a matrix vector equation).

While it is possible to find the roots of functions without using the derivative, those methods are not usually as fast or efficient as the Newton method. To take an advantage of it's speed and simplicity, you need to calculate the derivative of the function. In this example, we calculated the derivative by hand as it's not too hard to do. But what if $f$ had been much more complicated, and getting the derivative involved several pages worth of algebra? Well besides being a tedious calculation to do by hand, you often make mistakes! So it's important to check your work if you do all that algebra to make sure you didn't make a mistake somewhere. Or better yet, if you can automate the process of calculating the derivative, you can skip all that algebra to begin with.

Solving Optimization Problems¶

In many cases in science or applied math, we want to find the $x$ that minimizes a function $f(x)$ (or maximizes $-f(x)$). A good example is fitting curves or statistical models to data. These involve finding the minimum of a function (Note that the $x$ here isn't the $x$ in most statistical models. Statistical models use $x$ for data such as covariates rather the parameters $\theta$ which are the inputs we want to optimize for).

There are a number of different numerical algorithms for finding the solution to a minimization problem. Note that many algorithms assume a minimization problem, but you can convert a mazimization problem into a minimization problem by multiplying a function by $-1$. I won't go into the details of how these algorithms work, but I want to illustrate how algorithms which make use of the derivative of a function tend to be faster and more efficient computationally speaking than those that don't make those assumptions.

A lot of the fastest methods for optimization are based on the Newton method as discussed above. And these are called Newton methods. They require the first derivative (often called the gradient or jacobian) and sometimes also the second derivative (often called the hessian). A few methods will automatically approximate the first derivative using finite differences (more on this later), and these are called Quasi-Newton methods. They aren't as fast as the true Newton methods. Then there are non-Newton methods. Note we are only talking about local solution methods and not global ones which tend to not use derivative information at all (except that they might use a local minimization solver as part of their solution method). Maybe in the future we will do a post about these different solving techniques, but for now, we will use the methods provided by the scipy package in python to illustrate them. Methods also differ in whether or not they can deal with constraints (bounds/box, equality, inequality constraints)

First, though we need a function to minimize. We will use the Rosenbrock function, which is automatically provided for us by scipy. The Rosenbrock function is one of several benchmark functions that are used to test the capabilities of optimization solution methods (you can try one of the other functions in the wikipedia list if you'd like). It's rather easy to code too and it can be generalized to any dimension of input. presents some challenges for solvers.

Here's the definition for any $n$ dimensions : $$ f(x) = \sum_{i=1}^{n-1} 100\left(x_{i+1} - x_{i}^2 \right)^2 + \left(1 - x_i\right)^2 $$

The two dimension version is helpful for illustrating it's shape.

$$ f(x) = 100(x_2 -x_1^2)^2 + (1 + x_1)^2 $$

The Rosenbrock function always has a global minimum at $x = (1,1,\ldots,1)$ and for dimensions four through seven, it has a second minimum at $x = (-1,1,1,\ldots,1)$.

Automating the Calculation of Derivatives¶

If you took a calculus class, you almost certainly spent a number of homework assignments simply computing derivatives of all kinds of functions. And you could string these together using the chain rule and product rule and other ways for calculating derivatives. So you might be pretty good at calculating derivatives by hand. However, if you are good at calculating derivatives, it's not hard for functions to require several pages of algebra to get that derivative by hand. Even if you do that, it's easy to make a mistake (and using the wrong derivative is worse that not using a derivative at all). So at the very least, having a way to approximate the derivative would be useful to check our work by hand. And in many cases that approximation by might good enough.

Finite Differences¶

The first way we'll talk about is very simple to implement and is fairly useful. Take the Taylor series expansion for $f(x+h)$ and solve $f'(x)$:

$$ \begin{align} f(x+h) &= f(x) + f'(x)h + \frac{f''(x)}{2} h^2 + \cdots\\ \frac{f(x+h) - f(x)}{h} &= f'(x) + \frac{f''(x)}{2} h^2 + \cdots\\ \frac{f(x+h) - f(x)}{h} &\approx f'(x) \end{align} $$

Note this is the same as the limit definition of the first derivative! This is called the (forward) finite difference. To approximate the function $f$, we need to take a small step to a nearby input, calculate the function at that point, then divide by the difference. The advantage of this method is you don't have to know anything about the internal structure of $f$ to get a estimate of its derivative. When solvers for optimization approximate derivatives, they use some variation on this technique (note you should never provide a method that assumes an exact derivative with an finite-difference estimate, there are special methods that handle better the associated error!). This approximation is also used in solving certain partial differential equations too!

There are a few other finite difference approximations. The backward and central difference for example. I won't show them here, the central difference is a bit more accurate.

Note that the finite difference method works for higher dimensions too. For each partial derivative you take a step jsut in that direction. Letting $e_0,e_1,e_2,\ldots$ be the standard basis (i.e., $e_1 = (0,1,0,0,\ldots)$). $$ \frac{\partial f}{\partial x_i} \approx \frac{f(x+h*e_i) - f(x)}{h} $$

The scipy package provides a function that automatically calculates this for a given function. Let's try it on the Rosenbrock function and compare the results!

It's not because we did something wrong mathematically. The error continues to decrease mathematically speaking, but the computer suffers from a second type of error: cancellation error. As $h$ gets very small, the difference $(f+h) - f(x)$ gets very small too. The computer only represents real numbers with so many digits and beyond that level of precision, it cannot exactly represent a real number, meaning that there are errors in this substraction. In the extreme, once $h$ is small enough, $f(x+h)$ is so close to $f(x)$ that they only differ somewhere in the 10+ decimal places. The computer ends up get a difference of zero rather than the correct amount. So we end up estimating a derivative of zero. You can see this happens above at $h \approx 10^{-15}$.

Luckily, we will discuss a method for approximating derivatives that circumvents this error!

Symbolic Differentiation¶

It is possible to teach a computer to compute derivatives in the same way that you compute them on a piece of paper. The python package Sympy or SageMath and other Computer Algebra Systems can often take a written expression and manipulate it symbolically in a systematic way.

I am not going to go into detail here. In practice, this method isn't particularly flexible. It's also initially very slow (although you end up with an exact derivative at the end, which may not be slow to compute). But it's an option. The reason to mention symbolic differentiation is to contrast it with automatic differentiation which sounds similar but is very different in implementation.

Automatic Differentiation¶

Automatic differentiation (also sometimes called algorithmic differentiation or computational differentiation) is a method for automating the calculation of derivatives. Unlike finite differences, automatic differentiation is much more accurate (to within machine precision) and does not suffer from the numerical problems that finite difference approximations do. Unlike symbolic differentiation where the computer takes a string of symbols and inteprets them using algorithims to manipulate the symbols, automatic differentiation is essentially a way of differentiating computer code directly.

Of course, I haven't told you exactly what automatic differentiation is yet. As one of the references I read for this piece stated, there's actually a number of ways of coming up with the basic ideas, making it seem more complicated than it is. We're going to start with dual numbers as a simple version, partly because I think it's easiest to see how to implement dual numbers. We will use a simple method to approximate dual numbers using complex numbers.

Dual Numbers¶

Dual numbers are a lot like complex numbers. A complex number $z$ is composed of a pair of real numbers where one represents a real part and one represents the imaginary part. In other words, to make a complex number, you take the real numbers and you add in a special unit, called the imaginary unit $i$, with the rule that $i^2 = -1$. In general, you can write any complex number $z = x + iy$. From here, you can define arithmetic and a lot of calculus of real numbers on complex numbers. For instance, you can add, subtract, multiply, and divide as well as exponentiate, take logarithms, and extract roots! You can compute $\cos(z)$ and other transcendental functions too! It might not be obvious how to define those functions for complex numbers, but don't worry about that now. Things work out nicely for complex numbers.

Dual numbers are like complex numbers in that we start with the real numbers and add something. Instead of adding an imaginary unit $i$ with the defining rule $i^2 = -1$, we add a dual unit $\varepsilon$ with the rule $\varepsilon^2 = 0$. So a dual number is in general written as the pair $x + \varepsilon y $. Now operations and arithmetic are defined on dual numbers in the same way that we do for complex numbers. Let $z = x + \varepsilon y$ and $ w = u + \varepsilon v$ be two dual numbers.

Addition: $$ z + w = \left(x +u\right) \varepsilon \left(y + v\right) $$

That's what you'd expect. Things are a bit more interesting for multiplication. To multiply, we just multiply everything out as in regular algebra, and then collect the terms.

Multiplication: $$ zw = \left(x + \varepsilon y\right)\left(u + \varepsilon v\right) = xu + \left(xv + yu\right)\varepsilon + yv \varepsilon^2 = xu + \left(xv + yu\right)\varepsilon $$ The last equality follows from the rule that $\varepsilon^2 = 0$. Note that for the same reason, $\varepsilon^3,\varepsilon^4, \varepsilon^5,\ldots = 0$ because each term has a factor of $\varepsilon^2$.

So why does this help us compute derivatives? Let's look at an example with $f(x) = x^2$. If we apply $f$ to a dual number $x + \varepsilon$, what do we get?

$$ f(x + \varepsilon) = (x + \varepsilon)^2 = x^2 + 2x\varepsilon + \varepsilon^2 = x^2 + 2x\varepsilon $$

So we get another dual number, but if we split it into its real part and its dual part, we get $x^2$ and $2x$ which are the values of $f(x)$ and its derivative $f'(x)$! Does this work in general? Yes, let's take the Taylor serise expansion (technically we need to show that the Taylor series still works for dual numbers the way it does for real numbers, but we're going to lazy and informal and skip that rigorous proof). We'll pretend that the dual unit is just a real number to do the expansion then see what we get.

$$ f(x + \varepsilon) = f(x) + f'(x)\varepsilon + \frac{f''(x)}{2} \varepsilon^2 + \cdots $$

but the powers of $\varepsilon$ greater than 1 are all zero! So in fact, $$ f(x + \varepsilon) = f(x) + f'(x)\varepsilon $$ meaning that a function applied to a dual number is another dual number where the real part is the function applied to the real part and the dual part is the derivative of $f$ applied to the real part! In fact, to make this rigorous, we can actually use this as the definition $f$ on dual numbers. What I mean is that if you have a function $f$ on real numbers, and you want to convert it to a function on dual numbers, you can use the Taylor series expansion to define what $f$ should be on dual numbers (you can do something similar for complex numbers actually).

For example, what's $\sin(z)$ for a dual number $z = x + \varepsilon y$? Using our definition, $\sin(z) = \sin(x) +\varepsilon y \cos(x)$ because $\cos(x)$ is the derivative of $\sin(x)$ on real numbers.

So all operations we apply to dual numbers this way, we automatically have to compute the derivative as part of the definition. How does this automate the computation of derivatives? A really complicated function $f$ might compute a bunch of intermediate variables with simpler functions. E.g., maybe $ f(t) = f(x(t), y(t))$ where you compute some intermediate values $x$ and $y$ then use them in some computation. What happens when we apply this function to a dual number $t + \varepsilon$? The arithmetic of dual numbers automatically tracks and computes the chain rule and any other derivative combination rule. Here's the derivative of our example:

$$ \frac{df}{dt] = \frac{\partial f}{\partial x} \frac{\partial x}{\partial t} + \frac{\partial f}{\partial y} \frac{\partial y}{\partial t} $$

So when we apply those functions to dual numbers, what happens?

$$ x(t + \varepsilon) = x(t) + \varepsilon x'(t) \quad y(t + \varepsilon) = y(t) + \varepsilon y'(t) $$

and then applying $f$ to both of these dual numbers (using the Taylor series expansion as our definition): $$ f(x(t) + \varepsilon x'(t), y(t) + \varepsilon y'(t)) = f(x,y) + \varepsilon \left(\frac{\partial f}{\partial x}x' + \frac{\partial f}{\partial y}y'\right) $$

So if you were to define dual numbers in a computer and define functions on dual numbers, then anytime you wanted to compute a function on real numbers and its derivative, you simply use the dual number version. Basically, you are telling the computer to compute the derivatives for each computation along the way, no matter what computation it does. The arithmetic of dual numbers automates the rules of differentiation. Unfortunately, this isn't the easiest method to plug-and-play with, since you still need to define what dual numbers are and how to apply functions to them (which means you still have to tell the computer the derivative of everything elementary function). But once you do this, you now have a very flexible and fast way to compute derivatives of any function made up of those elementary functions.

In essence, all automatic differentation methods work this way. Typically they cannot handle functions which are not built from elementary functions that have derivatives defined in the library. The advantage is in quickly and accurately computing derivatives exactly where it would be hard to do by hand. Note that if we implement dual numbers into a computer, the derivatives would be computed as exactly as they could be if we had coded them by hand (to within machine precision, which means that any error results from the floating point arithmetic error only).

The Complex Step Method¶

Since we don't have easy implementations of dual numbers already to use in python, we will instead use complex numbers to approximate the same thing. This is called the complex step method. The idea is that we add a very small step in the imaginary unit direction to a real argument $f(x + ih)$ where $h$ is very small. Let's expand this as Taylor series (don't worry Taylor series expansion work for all complex functions with derivatives so we don't need to worry about whether or not this expansion is valid):

$$ f(x + ih) = f(x) + f'(x)ih + \frac{1}{2}f''(x) (ih)^2 + \frac{1}{3!}f'''(x) (ih)^3 + \cdots $$

So we get a complex number $ w = f(x+ih)$ where the real part is all the even power terms and the imaginary part is all of the odd powers $$ \begin{align} \mathrm{Re}(w) &= f(x) - \frac{1}{2}f''(x) h^2 + \frac{f^{(4)}(x)}{4!}h^4 +\cdots\\ \mathrm{Im}(w) &= f'(x)h - \frac{1}{3!}f'''(x) h^3 + \frac{f^{(5)}(x)}{5!}h^5 \cdots \end{align} $$ The first terms of each of these series is the function's value and the function's derivative times are our step $h$, plus a number of other terms. We can approximate the derivative thus by computing $$ f'(x) \approx \frac{1}{h}\mathrm{Im}\left(f(x + ih)\right) $$ We can also compute $f(x) \approx \mathrm{Re}(f(x+ih))$ meaning you only have run through the calculations of $f$ once to get both the function's value and its derivative at that point. Though you do have to do more calculations because at each line of $f$, you use a complex number instead a real number.

What a minute, didn't I say that this would be an exact method? Why is this any different than the finite differences? Unlike finite differences, we don't suffer from that cancellation error here! We are not subtracting anything, just adding a very small bit and then dividing by it. This means that we don't lose any accuracy as $h$ gets smaller! Also, for the same step size $h$, this method converges much faster! The error in approximation depends on the square $h^2$ rather than $h$ for the forward finite difference. Let's illustrate that numerically!

Differentiating the Farquhar-von-Caemmerer-Berry model of photosynthesis¶

After a lot of research, plant scientists were able to summarize the response of leaf photosynthesis to light and CO2 with a relatively simple photosynthetic function. However, this function isn't easy to differentiate. I'll show you this function and what it means, then we'll apply the complex step method to calculate it's derivative!

Briefly, photosynthesis has two series of reactions. A set of light reactions (which require light) and a set of carbon reactions (which require CO2). The light reactions strip electrons off water and use these electrons "charge" molecules (ATP, NADPH, NADH, etc.) which can take CO2 and fix it to sugar. The carbon reactions take CO2 molecules and bind them with a molecule called ribulose 1,5-bisphosphate (RuBP) via an enzyme called ribulose 1,5-bisphosphate carboxylase-oxygenase (RUBISCO), the most abundant enzyme on the planet! The acceptor molecule RuBP has to be regenerated by the light reactions.

The Farquhar-von-Caemmerer-Berry model thus approximates these two biochemical reaction pathways as a two rates of potential photosynthesis. (1) The rate of photoysnthesis (technically the rate of carboxylation) as limited by the regeneration of RuBP by the light reactions, denoted $A_j$. (2) The rate of photosynthesis as CO2 is limiting but RuBP is plentiful, denoted $A_c$. The observed photosynthesis is whichever is the lower of these two rates (the minimum) $A = \min\{A_j,A_c\}$.

The key equations are $$ \begin{align} A_c &= V_{c\max} \frac{C_c - \Gamma^*}{C_c + K_m} & &(\text{CO2 Limited Carboxylation Rate})\\ A_j &= \frac{J}{4}\frac{C_c - \Gamma^*}{C_c + 2\Gamma^*}& &(\text{RuBP Regeneration Rate})\\ J &= \frac{(J_\max+ \alpha I)+\sqrt{(J_\max+ \alpha I)^2 - 4\theta\alpha J_\max I}}{2\theta} & &(\text{Electron Transport Rate}) \end{align} $$

Most of the symbols here are parameters. The key variables are $C_c$ which is the CO2 concentration inside the chloroplasts and $I$ which is the light (photosynthetically active radiation; PAR). The third equation is an empirically derived equation that describes the response of the electron transport rate to light. This model has been tested numerous times in experiments and it's a very good approximation for photosynthesis in most plants. You can read more about the parameters here. Note that they use slightly different symbols for the same number than I did. The only constants not explained in the link are $\alpha$ which is effective absorption of light (sometimes called quantum yield) usually around 0.3-0.4. And $\theta$ which is just an empirical coefficient that controls the curvature of the electron transport rate's response to light. It also must between $[0,1]$ and is usually high in the range 0.7-0.95.

We will assume the other variables are constant. Most of them actually vary with temperature (and can vary over longer periods of time for other reasons) but for simplicity, we'll stick to this version.

Why would we want to know the derivative of this function? One reason is because this function gives us the rate of photosynthesis once you know the CO2 concentration inside the chloroplasts of a plant cell. That concentration depends on the rate of photosynthesis and the rate at which CO2 diffuses into the leaf, which depends on the stomatal conductance. Gases like CO2 and water are exchange with the atmosphere via small holes on the bottom of most leaves. These holes have two cells (called guard cells) that can open and close the hole. Together the hole and the guard cells is a called a stoma, and stomata when plural. The plant uses these stomata to regulate the gain of CO2 and loss of water, so the actual photosynthesis isn't just what we get from the above function. We often have to solve for the internal CO2 by equating the above function with a diffusion function or by adding in into a stomatal response model and solving for either a root or a minimum. This is easier to do numerically, and as we showed above, finding optima and roots is easier if you have the derivative.

The other reason is if we want to estimate these parameters from data. That is also a minimization problem. In that case, we would want to treat the parameters as the variables with light and CO2 held constant. So we would calculate our derivative with respect to those parameters rather than with respect to the light or CO2. Either way, we can use the complex step method!

    return 0.25*J(I)*(C - Gamma)/(C + 2*Gamma)

def J(I):
    b = Jmax + alpha*I
    return 0.5*(b + np.sqrt(b**2 - 4*Jmax*alpha*I*theta))/theta

def A(C,I):
    a = Ac(C)
    b = Aj(C,I)
    return np.minimum(a,b)

def A_ad(C,I):
    C += 1e-20j
    a = Ac(C)
    b = Aj(C,I)
    if a.real < b.real:
        return a.real, a.imag*1e20
    else:
        return b.real, b.imag*1e20

That's all there is to it! The complex step method is great because it gets a lot of benefit of automatic differentiation without being too tricky to implement since complex arithmetic has already been defined for many, many functions in most programming languages. Python's dynamic type system makes the conversion really easy too!

But for certain applications, the complex step method made be too slow or inefficient compared to a dedicated library. To really go whole hog on automatic differentiation, look into getting a dedicated package for automatic differentiation. Most programming languages will have at least a few implementations of automatic differentiation. For example, tensorflow in Python uses automatic differentiation. I recommend exploring them rather than building yoUr own because of the work required to make them work well for lots of different functions.

In a pinch, you can always resort to the complex step method! I hope you enjoyed today's post! Check out the references below if you want to read a bit more about automatic differentation!
